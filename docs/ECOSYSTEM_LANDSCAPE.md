# Open-Source Multi-Agent Ecosystem Landscape

**Research date:** 2026-07-20
**Data cutoff:** 2026-07-20
**Discovery pool:** 435 deduplicated GitHub search results; 240 retained in the first local candidate snapshot
**Initial curation:** 101 projects across 20 populated categories

## Overview

Open-source multi-agent work now spans two connected ecosystems. The first is LLM-agent engineering: role-based teams, supervisors, handoffs, task graphs, shared context, coding teams, deep research, enterprise workflows, and agent interoperability. The second is the older multi-agent systems and multi-agent reinforcement learning ecosystem: negotiation, robot teams, social simulations, cooperative environments, safety, and benchmark suites.

Frameworks and coding systems remain the densest areas. Research automation, finance, data analysis, browser operation, and enterprise accelerators form a visible middle tier. Healthcare, education, legal, customer service, and marketing contain fewer mature open implementations; these areas are therefore curated conservatively rather than padded with demos.

## Major technical directions

- Role-based and hierarchical teams with managers, specialists, critics, and reviewers.
- Handoffs and decentralized swarms that transfer control between peers.
- Graph and event-driven runtimes for durable, concurrent execution.
- Protocols and meshes for agent discovery, messaging, and task exchange.
- Shared memory, traceability, evaluation, and operational governance.
- Domain teams that separate research, planning, execution, validation, and synthesis.
- Multi-agent reinforcement learning environments for cooperation, competition, and emergence.
- Multi-robot planning, safe control, and distributed embodied coordination.

## Density by domain

The first release is intentionally uneven. Frameworks, coding, robotics, simulation, evaluation, and research have stronger open-source depth. Finance and enterprise have several substantial systems. Browser, cybersecurity, healthcare, education, legal, marketing, and customer service remain smaller and include some clearly marked experimental projects.

## Classification basis

A project receives exactly one primary category based on its central, mature use. Framework examples do not turn a framework into an application. Complete applications are classified by what they deliver, not by the SDK beneath them. Tags capture secondary coordination patterns or technical traits.

## Difference from general AI-agent lists

This directory does not equate “agentic” with “multi-agent.” A single agent using many tools, a pipeline with several nodes, ordinary RAG, or an MCP server does not qualify by itself. The review looks for independent agent responsibilities and an explicit coordination mechanism.

## Differentiated value

The project combines a strict boundary, domain-first browsing, bilingual editorial descriptions, one structured source for README and website output, explicit historical status, local untrusted-candidate handling, and a weekly PR-only update process. It favors long-term trust over maximum item count.

## Gaps

The largest gaps are mature open applications in regulated and customer-facing domains, shared-memory systems designed specifically for teams, multi-agent safety/governance beyond generic tracing, and comparable evaluation of real LLM-agent teams. Regional discovery also remains sensitive to language and repository naming, so future searches should rotate local-language terms and organization seeds.
